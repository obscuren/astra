# Remove Landing Pad POI, Add Ship-Tab Board Ship Action — Design Spec

**Date:** 2026-04-11
**Status:** Draft
**Scope:** Replace the physical `Tile::OW_Landing` POI with a UI-driven "Board Ship" action on the character screen's Ship tab.

## Context

Every planet overworld currently contains a single `Tile::OW_Landing` tile placed by `OverworldGeneratorBase::place_landing_pad()` at or near the map center. When the player arrives on a body, `game_world.cpp` spawns them on that tile; pressing `e` on it descends into a detail map that includes the ship's cockpit layout. This requires the player to physically walk to a landing pad before interacting with their ship, and it hard-codes an ugly special case into the overworld generator + detail map pipeline.

The player already has a fully walkable ship interior (`MapType::Starship`, four rooms generated by `starship_generator.cpp`) used when flying in deep space. The character screen (Tab) already has a `CharTab::Ship` tab showing components, diagnostics, and cargo, with a stub footer note "Board your ship to manage equipment." implying the player must physically be in the cockpit.

This spec replaces the landing-pad flow with a menu-driven boarding action: from any walkable overworld tile, Tab → Ship → `Board Ship` warps the player into `MapType::Starship` at the cockpit region. Exiting the ship restores the exact overworld tile they boarded from.

## Decisions Summary

| Decision | Choice | Rationale |
|---|---|---|
| Planet arrival spawn | Deterministic center-adjacent walkable tile | Reproducible across regenerations; matches where landing pad used to sit |
| Boarding scope | Overworld only | Matches the fiction ("your ship is parked on the planet"); dungeons force the player to climb out first |
| Ship tab UI | Dedicated `» Board Ship «` action row at the top of the Ship tab | Minimal UI change; structured for future actions (Repair, Launch, etc.) |
| Return-from-ship | Remember overworld (x, y) at board time, persist in save, restore on disembark | Natural "step out where you parked" behavior |
| What Board Ship does | Switches world to `MapType::Starship`, spawns player in Cockpit region | Reuses the existing ship interior; no second cockpit layout to maintain |
| `Tile::OW_Landing` enum | Keep as deprecated no-op | Avoids shifting enum values and breaking in-flight saves |
| Proximity check | None — board from any overworld tile | Simplifies UX; matches user intent |

## 2. Data Flow

### Deterministic center-adjacent spawn

Replaces the `OW_Landing` scan in `game_world.cpp:1215-1229`:

```cpp
std::pair<int,int> find_center_spawn(const TileMap& map) {
    int cx = map.width() / 2;
    int cy = map.height() / 2;

    // Walk outward in expanding rings until we find a walkable tile.
    for (int r = 0; r < std::max(map.width(), map.height()); ++r) {
        for (int dy = -r; dy <= r; ++dy) {
            for (int dx = -r; dx <= r; ++dx) {
                if (std::abs(dx) != r && std::abs(dy) != r) continue;
                int px = cx + dx, py = cy + dy;
                if (px < 0 || px >= map.width() ||
                    py < 0 || py >= map.height()) continue;
                Tile t = map.get(px, py);
                if (t == Tile::OW_Mountains || t == Tile::OW_Lake ||
                    t == Tile::OW_River    || t == Tile::OW_Swamp) continue;
                return {px, py};
            }
        }
    }
    return {cx, cy};
}
```

This mirrors the legacy `place_landing_pad()` search so the "arrival spot" matches where the landing pad used to sit — players who develop muscle memory about where things are should not notice a difference except that the glyph is gone.

### `OverworldReturnPos` on `World`

```cpp
struct OverworldReturnPos {
    bool valid = false;
    int x = 0;
    int y = 0;
    std::string body_key;  // matches current location key used by location_cache
};
```

- **Set** in `board_ship_from_overworld()` to the player's current overworld position + body key.
- **Cleared** in `disembark_ship_to_overworld()` after restoring the player.
- **Cleared** automatically when the player warps to a different body (detected by body_key mismatch on disembark — fall back to fresh-arrival path).
- **Persisted** in the save file as three fields (valid bool, x int, y int) + body_key string.

### `board_ship_from_overworld()`

```cpp
void Game::board_ship_from_overworld() {
    // Preconditions checked by caller via can_board_ship().
    // Save the return position.
    overworld_return_.valid = true;
    overworld_return_.x = player_.x;
    overworld_return_.y = player_.y;
    overworld_return_.body_key = current_location_key();

    // Cache the current overworld map so we can restore it on disembark.
    cache_current_location();

    // Load or generate the Starship map.
    auto props = default_properties(MapType::Starship);
    world_.map() = TileMap(props.width, props.height, MapType::Starship);
    auto gen = create_starship_generator();
    gen->generate(world_.map(), props, world_.seed());

    // Place player in the cockpit region.
    place_player_in_cockpit();

    world_.set_surface_mode(SurfaceMode::Dungeon);  // existing mode used for ship interiors
    world_.visibility() = VisibilityMap(world_.map().width(), world_.map().height());
    recompute_fov();
    compute_camera();
}
```

### `disembark_ship_to_overworld()`

Triggered from `game_interaction.cpp` when the player uses the cockpit's exit hatch and `overworld_return_.valid && overworld_return_.body_key == current_location_key()`:

```cpp
void Game::disembark_ship_to_overworld() {
    if (!overworld_return_.valid) return;

    // Restore the cached overworld map.
    restore_location(overworld_return_.body_key);

    player_.x = overworld_return_.x;
    player_.y = overworld_return_.y;

    overworld_return_ = {};  // clear

    world_.set_surface_mode(SurfaceMode::Overworld);
    recompute_fov();
    compute_camera();
}
```

If `overworld_return_.valid` is false or the body_key mismatches (player warped to a new body from inside the ship), fall through to the existing fresh-arrival flow.

### Boarding flow diagram

```
[Player on planet overworld]
        │ Tab pressed
        ▼
[CharacterScreen opens, focus on Ship tab action row]
        │ Enter on "Board Ship"
        ▼
[char_screen sets board_ship_requested_ = true, closes]
        │
        ▼
[Game::update sees consume_board_ship_request()]
        │
        ▼
[board_ship_from_overworld()
   - save overworld_return_
   - cache current map
   - generate MapType::Starship
   - place player in Cockpit region
   - FOV + camera]
        │
        ▼
[Player is in the ship interior]
```

### Disembark flow diagram

```
[Player in ship interior]
        │ steps on exit hatch / cockpit stairs
        ▼
[game_interaction detects ship exit]
        │
        ▼
[if overworld_return_.valid && body_key matches:
    disembark_ship_to_overworld()
 else:
    existing exit-ship logic (deep space)]
        │
        ▼
[Player back at saved overworld (x, y)]
```

### Validity check

```cpp
bool can_board_ship(const World& world) {
    if (world.map().map_type() != MapType::Overworld) return false;
    if (world.surface_mode() != SurfaceMode::Overworld) return false;
    if (world.navigation().at_station) return false;
    if (world.navigation().on_ship) return false;
    return true;
}
```

## 3. UI — Ship Tab Action Row

### Layout

Added at the top of the Ship tab, above `COMPONENTS`:

```
┌─────────────────── Ship ───────────────────┐
│ <ship name> (Class I)            Operational│
│                                             │
│ ACTIONS                                     │
│ ▸ Board Ship                                │  ← selected (when eligible)
│                                             │
│ COMPONENTS                                  │
│ Engine:     Coil Drive Mk1                  │
│ ...                                         │
└─────────────────────────────────────────────┘
```

When `can_board_ship()` is false:

```
│ ACTIONS                                     │
│   Board Ship — must be on a planet          │  ← TextDim, not selectable
```

Disabled reason strings:
- `"must be on a planet"` — when on a station, ship, or inside a dungeon.
- `"already on your ship"` — when already in the Starship map.

Only one string displayed at a time; pick whichever applies.

### New `CharacterScreen` state

```cpp
enum class ShipFocus { Actions, Equipment, Inventory };
int ship_action_cursor_ = 0;          // only 0 = Board Ship for now
bool board_ship_requested_ = false;

public:
  bool has_board_ship_request() const { return board_ship_requested_; }
  bool consume_board_ship_request() {
      bool r = board_ship_requested_;
      board_ship_requested_ = false;
      return r;
  }
```

### Input handling on Ship tab

When `ship_focus_ == ShipFocus::Actions`:
- **Up/Down** — navigates the action list (currently no-op, only 1 entry).
- **Tab / right-arrow** — moves focus to `Equipment`, then `Inventory`.
- **Left-arrow** from `Equipment` — moves focus back to `Actions`.
- **Enter** on an enabled action — sets `board_ship_requested_ = true` and closes the character screen.
- **Enter** on a disabled action — sets `context_message_` to the reason for a brief feedback flash.

When the Ship tab is first opened:
- If `can_board_ship()` is true → default `ship_focus_ = Actions` so keyboard-first users land on the Board Ship row.
- Else → default `ship_focus_ = Equipment` so the cursor doesn't sit on a disabled row.

### Game-side wiring

In `game.cpp` after `character_screen_.handle_input()`:

```cpp
if (character_screen_.consume_board_ship_request()) {
    board_ship_from_overworld();
    // Character screen already closed itself.
}
```

## 4. Edge Cases

1. **Player is in a dungeon on a planet.** `can_board_ship()` returns false. UI shows "must be on a planet" — they have to climb out first.
2. **Player is on a station (Heavens Above).** Disabled. They use the station's existing `ShipTerminal` fixture as before.
3. **Player is already on their ship.** Disabled with "already on your ship". Ship tab still shows components/cargo for management.
4. **Save mid-session in the ship after boarding from a planet.** `OverworldReturnPos` is serialized. Load + exit ship correctly restores the tile.
5. **Player warps to a different body from inside the ship.** On disembark, the body_key mismatches → fall back to the normal fresh-arrival path (spawn at `find_center_spawn()`), `overworld_return_` cleared.
6. **Combat on the overworld.** Boarding is allowed — it's a menu action, instantaneous. Future balance concern noted in out-of-scope.

## 5. Save File Changes

`OverworldReturnPos` adds three fields + a string. Version bump or additive serialization depending on how the save format currently handles new fields. Recommend additive: write the new fields at the end of the world-state block so old saves without them load with `valid = false`.

## 6. Testing & Verification

- [ ] Fresh game, travel to a planet → player spawns at the deterministic center-adjacent tile (no `OW_Landing` glyph visible anywhere on the overworld).
- [ ] Tab → Ship tab → action row shows `Board Ship`, selected and highlighted.
- [ ] Enter on Board Ship → player is inside their ship in the Cockpit.
- [ ] Exit ship via cockpit stairs → player is back at the exact overworld tile they boarded from.
- [ ] Walk around on overworld, board ship again from a different tile → exit → returns to that new tile.
- [ ] In a dungeon → Tab → Ship tab → Board Ship row shows "must be on a planet", not selectable.
- [ ] At Heavens Above station → Tab → Ship → Board Ship disabled. Existing station ShipTerminal still works as before.
- [ ] In deep space on the ship → Tab → Ship → Board Ship disabled.
- [ ] Save on planet → exit game → load → still on planet at the right tile, Board Ship works.
- [ ] Save in ship after boarding from a planet → exit game → load → still in ship → exit ship → back on the correct planet tile.
- [ ] Save in ship after boarding from a planet → warp to a different body → land → Board Ship return state was cleared correctly; spawn at new body's center-adjacent tile.
- [ ] Map editor no longer offers `OW_Landing` as a placeable tile.
- [ ] Dev POI stamp test menu no longer offers Landing Pad.

## 8. Out of Scope

- Proximity check for boarding (must-be-near-ship gating).
- Animation or cutscene for ship arrival or takeoff.
- Boarding from detail maps (surface zones). Only overworld boarding is enabled.
- Multiple action rows beyond `Board Ship` (Repair / Launch / Warp continue using existing flows).
- Changes to the station `ShipTerminal` or the deep-space flight flow.
- Combat turn cost for boarding.
- Preventing board during combat.
- Fully removing the `Tile::OW_Landing` enum value from `Tile::`.
- Landing pad dungeons (trivially none existed).
- Porting the legacy landing-pad detail-map stamp to v2 (being deleted instead).
